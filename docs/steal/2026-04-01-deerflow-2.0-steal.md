## P0 模式（立即可偷，直接提升架构）

### P0-1: Middleware Pipeline — 12 层链式 Agent 行为组合

**来源**: `backend/packages/harness/deerflow/agents/lead_agent/agent.py:208-270`

**模式**: Agent 的所有横切关注点（安全、记忆、循环检测、上下文压缩、子 agent 限流、Todo、Token 追踪等）通过 `AgentMiddleware` 链串联。每个 middleware 可以挂接 `after_model`、`after_agent`、`wrap_tool_call` 等生命周期钩子。

**12 层 middleware**:
1. ToolErrorHandling — 工具异常转 ToolMessage
2. SummarizationMiddleware — Token 触发上下文摘要
3. TodoMiddleware — Plan 模式下的任务追踪
4. TokenUsageMiddleware — 每次模型调用统计 token
5. TitleMiddleware — 首次交互后生成标题
6. MemoryMiddleware — 对话结束后入队记忆更新
7. ViewImageMiddleware — 注入图片描述（仅 vision 模型）
8. DeferredToolFilterMiddleware — 延迟加载工具隐藏
9. SubagentLimitMiddleware — 硬截断超限子 agent 调用
10. LoopDetectionMiddleware — 滑窗哈希检测重复调用
11. 自定义 middleware（可注入）
12. ClarificationMiddleware — 最后一层，拦截澄清请求

**核心设计**:
```python
# 顺序决定语义：错误处理先于压缩，压缩先于记忆，澄清永远最后
# ThreadDataMiddleware 必须在 SandboxMiddleware 之前（确保 thread_id 可用）
# MemoryMiddleware 在 TitleMiddleware 之后（标题生成后再存记忆）
```

**我们缺什么**: Orchestrator 的 Governor 是单体执行器，没有 middleware 管道。横切关注点（安全检查、token 追踪、循环检测）散落在不同层。

**偷法**: Governor 执行管道抽象为 middleware chain。每个关注点独立为 middleware，通过 `before_tool` / `after_model` / `after_agent` 生命周期钩子注入。Priority: 用 Python Protocol 定义接口，已有的 guard.sh 功能迁移为第一个 GuardMiddleware。

### P0-2: LoopDetection — 滑窗哈希 + 警告→硬停双阶段

**来源**: `backend/packages/harness/deerflow/agents/middlewares/loop_detection_middleware.py`

**模式**: 每次模型输出后，对 tool_calls 做 `(name, args)` 排序后 MD5 哈希，存入 per-thread 滑窗（默认 20）。相同哈希出现 ≥3 次注入警告，≥5 次直接剥离 tool_calls 强制文本输出。

**关键细节**:
- **OrderedDict LRU 淘汰**: 最多追踪 100 个 thread，超限淘汰最旧的
- **警告用 HumanMessage 而非 SystemMessage**: Anthropic 模型禁止中途插入 SystemMessage（#1299 bug）
- **硬停不是抛异常**: 而是 `model_copy(update={"tool_calls": []})` 替换消息，保留已有内容

**我们缺什么**: Orchestrator 没有循环检测。Agent 陷入重复调用只能靠 recursion_limit 硬杀。

**偷法**: Governor 的 tool 执行管道加入 loop detector。我们不需要 LangGraph middleware 接口，直接在 `execute_tool` 前后加 hash 追踪即可。警告阈值和硬停阈值可配置。

### P0-3: Memory Update Queue — 异步去抖 + LLM 驱动结构化更新

**来源**:
- `backend/packages/harness/deerflow/agents/memory/queue.py` — 去抖队列
- `backend/packages/harness/deerflow/agents/memory/updater.py` — LLM 驱动更新
- `backend/packages/harness/deerflow/agents/memory/prompt.py` — 记忆注入 prompt

**模式**: 三层记忆管道:
1. **MemoryMiddleware** (after_agent) — 过滤对话消息（只保留 user + final AI response，剥离工具调用中间步骤），入队到单例 MemoryUpdateQueue
2. **MemoryUpdateQueue** — 同一 thread 重复入队时替换旧版本（去重），Timer 去抖（默认 30s），daemon 线程异步处理
3. **MemoryUpdater** — 用 LLM 分析对话，输出结构化 JSON 更新

**记忆结构**:
```json
{
  "user": {
    "workContext": {"summary": "...", "updatedAt": "..."},
    "personalContext": {"summary": "...", "updatedAt": "..."},
    "topOfMind": {"summary": "...", "updatedAt": "..."}
  },
  "history": {
    "recentMonths": {"summary": "..."},
    "earlierContext": {"summary": "..."},
    "longTermBackground": {"summary": "..."}
  },
  "facts": [
    {"id": "fact_xxx", "content": "...", "category": "preference|knowledge|context|behavior|goal", "confidence": 0.9, "source": "thread_id"}
  ]
}
```

**独特设计**:
- **Upload Mention Stripping**: 正则自动剥离 `<uploaded_files>` / `/mnt/user-data/uploads/` 相关事实，防止临时文件路径污染长期记忆
- **Fact Dedup**: 新 fact 入库前做 content 去重（strip + exact match）
- **Max Facts 溢出淘汰**: 超 `max_facts` 限制时按 confidence 降序保留
- **Token-Aware 注入**: 记忆注入 system prompt 时用 tiktoken 精确计数，facts 按 confidence 降序填充到 token 预算用完

**我们缺什么**: Orchestrator 的 memory 是追加式文件（experiences.jsonl + 手动 memory 文件），没有自动去抖更新、没有结构化分区、没有 fact confidence 排序。

**偷法**:
1. experiences.jsonl → 6 分区结构化记忆（user 3 区 + history 3 区 + facts）
2. 对话结束后异步 LLM 抽取 → debounce queue → 合并更新
3. 记忆注入 boot.md 时做 token 预算控制

### P0-4: GuardrailProvider Protocol — 可插拔工具调用授权

**来源**:
- `backend/packages/harness/deerflow/guardrails/provider.py` — Protocol 定义
- `backend/packages/harness/deerflow/guardrails/middleware.py` — Middleware 集成
- `config.example.yaml:653-686` — 三种 Provider 选项

**模式**: 每个 tool call 执行前通过 `GuardrailProvider.evaluate(request)` 授权。Provider 是 Protocol（不是基类），任何实现了 `evaluate`/`aevaluate` 的类都能用。

**三层 Provider 体系**:
1. **AllowlistProvider** (内置) — denied_tools 列表，零外部依赖
2. **OAP Provider** — Open Agent Passport 标准，任何 OAP 实现
3. **Custom Provider** — 任何类路径，通过 `resolve_variable()` 动态加载

**GuardrailMiddleware 设计**:
```python
# fail_closed=True: Provider 异常时默认拒绝（安全优先）
# 拒绝不是抛异常: 而是返回 ToolMessage(status="error")，Agent 可以自适应
# GraphBubbleUp 信号透传: LangGraph 的 interrupt/pause/resume 不被拦截
```

**我们缺什么**: guard.sh 是正则静态拦截 + audit.sh 审计。没有 Protocol 接口，没有 fail-closed 语义，没有 Agent 自适应能力（拦截就是死）。

**偷法**: 定义 `GuardProvider` Protocol，guard.sh 重构为 `RegexGuardProvider`，未来可叠加 LLM 语义评估层。被拒绝时 Agent 收到错误消息而不是进程崩溃。

### P0-5: SubagentLimitMiddleware — 硬截断超限并行调用

**来源**: `backend/packages/harness/deerflow/agents/middlewares/subagent_limit_middleware.py`

**模式**: LLM 生成的 tool_calls 中，如果 `task` 调用超过 `max_concurrent`（默认 3，上限 4），middleware 直接截断多余的 tool_calls。不是靠 prompt 约束——prompt 里写了限制，但 middleware 做硬执行。

**关键洞察**: "Prompt-based limits are unreliable; middleware enforcement is more reliable"。

**我们缺什么**: Orchestrator 的 sub-agent 调用没有并发限制中间件。Agent SDK 的 `max_turns` 只限总轮数，不限每轮并发。

**偷法**: Governor 的 agent 调用管道加入并发限制。每轮 AI 输出中超过 N 个 tool_calls 时截断尾部。

### P0-6: DeerFlowClient — 嵌入式 Python 客户端

**来源**: `backend/packages/harness/deerflow/client.py`

**模式**: 同一套 Agent 逻辑既能跑 HTTP 微服务（LangGraph Server），也能 in-process 调用（`DeerFlowClient`）。client 提供 `chat()`、`stream()`、`list_models()`、`list_skills()`、`get_memory()` 等完整 API。

**关键设计**:
- **Lazy Agent Creation**: Agent 在首次调用时创建，config 变化自动重建（通过 config key tuple 比较）
- **StreamEvent 协议兼容**: 嵌入式事件流和 LangGraph SSE 协议一致，消费端无需改代码
- **Atomic JSON Write**: `tempfile + replace` 原子写入，防止并发损坏

**我们缺什么**: Orchestrator 的 Agent 只能通过 Docker + CLI 调用。没有 in-process 嵌入模式。

**偷法**: 抽象 `OrchestratorClient` 接口，Governor 的执行逻辑既能被 Docker 容器调用，也能被 Python 脚本直接 import。CI 测试用嵌入模式，生产用容器模式。

## P1 模式（重要但非紧急）

### P1-1: Configurable Summarization — 多触发源上下文压缩

**来源**: `config.example.yaml:496-538`, `agent.py:41-80`

**模式**: 上下文压缩不是硬编码触发，而是可配置多种触发源:
- `tokens`: 固定 token 数（如 15564）
- `messages`: 消息数量（如 50 条）
- `fraction`: 模型最大输入的百分比（如 80%）

可以同时配置多个触发器（OR 逻辑），加上可配置的 `keep` 策略（保留最近 N 条/N tokens/N%）。

**偷法**: Governor 的 context 管理从硬编码 token 限制改为可配置触发器 + 保留策略。

### P1-2: Upload Mention Stripping — 临时文件记忆清洗

**来源**: `updater.py:209-240`

**模式**: 用正则从记忆的 summary 和 facts 中主动剥离文件上传相关内容（`/mnt/user-data/uploads/`、`<uploaded_files>` 等）。因为上传文件是 session 级的，在长期记忆中保留会导致 Agent 在未来 session 搜索不存在的文件。

**偷法**: memory updater 加入 ephemeral content stripper。我们的场景：Docker 容器路径、临时文件路径不应进入长期记忆。

### P1-3: Fact Confidence Ranking — 记忆注入时按置信度排序

**来源**: `prompt.py:186-294`

**模式**: 记忆注入 system prompt 时，facts 按 confidence 降序排列，逐条累加 token 直到预算用完。低置信度 fact 自然被截断。非有限数值（NaN, inf）安全处理为 default。

**偷法**: 我们的 memory 注入加入 token 预算 + 优先级排序。高频/高确定性记忆优先进 context。

### P1-4: Skill Progressive Loading — 仅在需要时读取

**来源**: `prompt.py:383-424`, `loader.py`

**模式**: Skills 不是全量注入 system prompt。只列出 `<name>` + `<description>` + `<location>`，Agent 匹配到对应 skill 时用 `read_file` 加载完整内容。

**与我们的区别**: 我们的 skill 全文直接编译进 system prompt（context pack），DeerFlow 是延迟加载。他们的方法节约 context 但增加一次 tool call。

**偷法**: 对于大型 skill（> 2K tokens），改为 progressive loading。小 skill 继续 inline。

### P1-5: Clarification-First Workflow — 澄清优先于执行

**来源**: `prompt.py:178-246`

**模式**: System prompt 强制 "CLARIFY → PLAN → ACT" 工作流。5 种澄清类型（missing_info, ambiguous_requirement, approach_choice, risk_confirmation, suggestion），配合专用 `ask_clarification` 工具，调用后自动中断执行。

**偷法**: Governor 的 task 派发前加入 clarification 检查节点。当任务描述模糊时，先问再做。

### P1-6: Per-Agent Memory — 多 Agent 独立记忆空间

**来源**: `updater.py:269-276`, `middleware.py:102-109`

**模式**: 每个 named agent 可以有独立的 memory storage（通过 `agent_name` 参数路由）。全局 agent 用共享记忆，专业 agent 用独立记忆。

**偷法**: 三省六部各部门可以有独立记忆空间（如工部的技术记忆、礼部的沟通记忆）。

## P2 模式（参考价值）

### P2-1: Soul System — Agent 人格注入

**来源**: `prompt.py:427-432`, `config/agents_config.py`

Agent 支持 `SOUL.md` 人格文件，格式: `<soul>...</soul>` XML 包裹注入 system prompt。跟我们的 SOUL 系统理念一致，但他们是 per-agent 的。

### P2-2: ACP Agent Integration — 外部 Agent 互操作

**来源**: `config.example.yaml:437-463`, `tools/builtins/invoke_acp_agent_tool.py`

通过 ACP (Agent Communication Protocol) 调用外部 agent（Claude Code、Codex CLI），结果写入 `/mnt/acp-workspace/`。这是 multi-agent 互操作的标准化尝试。

### P2-3: Deferred Tool Loading — 延迟工具发现

**来源**: `prompt.py:435-457`, `tools/builtins/tool_search.py`

MCP 工具不直接加载到 context，而是列出名称供 `tool_search` 发现。减少 context 膨胀。跟 Claude Code 的 ToolSearch 完全一致。

### P2-4: Dual ThreadPool Executor — 调度与执行分离

**来源**: `subagents/executor.py:71-76`

子 agent 用两个 ThreadPool: `scheduler-pool`（3 worker）负责编排, `execution-pool`（3 worker）负责实际运行。调度线程提交到执行线程并设 timeout，超时可取消。避免调度阻塞。

### P2-5: IM Channel Auto-Threading — 全 inbound 无需公网

**来源**: `config.example.yaml:588-651`, `app/channels/`

Telegram(polling), Slack(Socket Mode), 飞书(WebSocket) 全部用 inbound 连接，无需公网 IP 或 webhook。每条消息自动关联 LangGraph thread_id。

### P2-6: Polymorphic Class-Path Resolution — `use` 字段统一组件加载

**来源**: `config.example.yaml`, `reflection/resolvers.py`

所有可插拔组件（模型、工具、沙箱、guardrail）统一用 `use: "module.path:ClassName"` 格式，运行时 `resolve_class()` 动态 import。模型配置用 Pydantic `extra="allow"`，YAML 参数直接透传给 LangChain 构造器，不用为每个 provider 写专属 schema。

### P2-7: SandboxAuditMiddleware — Bash 命令三级分类

**来源**: `agents/middlewares/sandbox_audit_middleware.py`

Bash 命令按危险程度分三级：`block`（`rm -rf /`、`curl|sh`、`dd if=`）直接返回错误 ToolMessage，`warn`（`pip install`、`chmod 777`）执行但追加警告，`pass` 放行。每个调用都有结构化 JSON 审计日志。用 regex + shlex 双重解析检测。

### P2-8: Anchor-Based Middleware Insertion — `@Next`/`@Prev` 锚点

**来源**: `agents/features.py:41-60`

第三方 middleware 可以声明 "我要插在 X 之后/之前"（`@Next("LoopDetection")`），不需要知道链的完整索引。工厂方法做迭代解析，支持跨 extra-to-extra 锚定，检测循环依赖。配合三态 Feature Toggle（True=内置默认 / False=禁用 / AgentMiddleware 实例=自定义替换）。

### P2-9: Config Hot-Reload via mtime — 零成本配置热重载

**来源**: `config/app_config.py:269-294`

`get_app_config()` 是带 mtime 检测的 singleton。每次调用检查文件修改时间，变了就重载。跨进程配置同步不用 IPC，读 mtime 就行。

## 与 Orchestrator 的对比分析

| 维度 | DeerFlow 2.0 | Orchestrator | 差距 |
|------|-------------|-------------|------|
| **Agent 行为组合** | 12 层 Middleware Pipeline + 锚点插入 | 单体 Governor | 🔴 缺 middleware 管道 |
| **循环检测** | 滑窗哈希 + 双阶段 | 无 | 🔴 完全缺失 |
| **记忆系统** | 6 分区结构化 + LLM 驱动 + debounce | 追加式 jsonl + 手动 memory 文件 | 🔴 架构差距大 |
| **工具授权** | GuardrailProvider Protocol + Audit 三级 | guard.sh 正则 | 🟡 功能有但不可编程 |
| **子 agent 限流** | Middleware 硬截断 | 无 | 🟡 靠 prompt 约束 |
| **上下文压缩** | 可配置多触发源 | 硬编码 | 🟡 功能有但不灵活 |
| **嵌入式客户端** | DeerFlowClient (isomorphic API) | 无 | 🟡 不紧急 |
| **组件加载** | `use` 字段 + resolve_class() 动态加载 | 硬编码 import | 🟡 灵活性差距 |
| **配置热重载** | mtime 检测自动重载 | 需要重启 | 🟡 不紧急 |
| **Skill 系统** | SKILL.md + progressive load | SKILL.md + context pack | 🟢 接近，他们更省 context |
| **IM 通道** | Telegram + Slack + 飞书 | Telegram | 🟢 功能够用 |
| **SOUL 人格** | per-agent soul | 全局 SOUL 系统 | 🟢 我们更成熟 |
| **治理架构** | 无 | 三省六部 | 🟢 我们独有 |
