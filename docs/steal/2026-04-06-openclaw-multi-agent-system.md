## 背景

OpenClaw（又名 Clawdbot）是一个基于 Claude Code 定制的多 Agent 系统，面向**工控项目管理**场景。作者 "徐Sir" 把真实工控项目的组织架构（PM → 硬件/采购/软件/调试）映射成了 AI Agent 角色，用飞书作为沟通渠道。

系统的底座是 Claude Code 的 workspace/agent 机制，但上层设计有自己的思路——特别是角色约束、沟通协议和分层审核方面。

---

## 架构概览

```
老板 (人)
  │ sessions_send (私聊派活)
  ▼
PM-AI员工 (work01)
  ├── 创建项目群聊会话，邀请各角色
  ├── 群聊中发布任务，共享目录放文档
  ├── 每日群聊站会，各代理汇报状态
  ├── 深度协作用 sessions_send 私聊
  └── 汇总后 message.send → 人
  │
  ├─ Hardware (work02)    → 02_电气设计/
  ├─ Procurement (work03) → 03_采购/
  ├─ Software (work04)    → 04_程序/
  └─ Commissioning (work05) → 05_文档/
```

**数据拓扑**: 每个 Agent 有私有 workspace（notes/ 不共享），通过符号链接共享 `shared-projects/{项目名}/` 目录。这是唯一的数据源。

## 偷师模式清单

### P1: 结构化角色约束文件 ⭐⭐⭐

每个 Agent 的 `role.md` 用固定 5 段结构定义角色边界：

```markdown
# 角色约束 - {角色名}

## 我是谁
- **角色:** 硬件工程师
- **上级:** 项目经理
- **协作:** 采购工程师、PLC 工程师

## 我的职责
1. 阅读项目需求文档
2. 设计电气原理图
3. 输出 BOM 清单（含型号、数量、供应商）
4. 输出 IO 分配表
5. 配合采购确认替代型号

## 沟通规范
| 场景     | 渠道              | 说明                  |
|----------|-------------------|-----------------------|
| 任务确认 | 私聊 (PM→HW)      | 接收任务细节          |
| 每日站会 | 群聊              | 汇报进度、阻塞        |
| BOM 完成 | 群聊 + 私聊        | 群聊通报，私聊采购    |
| 技术讨论 | 私聊 (HW↔SW)      | IO 分配、接线确认      |

## 文件规范
- **私有笔记:** `{workspace}/notes/`（设计思路、计算过程）
- **项目输出:** `shared-projects/{项目名}/02_电气设计/`（通过符号链接）

## 禁止行为
- 不代替其他代理做专业决策
- 不直接修改其他代理的输出文件
- 不在群聊中泄露敏感信息（预算、人员）
```

**对比 Orchestrator**: 我们的三省六部 department prompt 是纯自然语言，没有这种表格化的沟通协议。这个格式更机器友好、更不容易遗漏。

**可偷**: 给六部的 prompt 加上结构化的「沟通规范」表——明确哪种场景走哪个通道、跟谁协作。

### P2: Agent 沙箱权限模型 ⭐⭐⭐

每个 Agent 的 `config.json` 有 constraints 字段：

```json
{
  "constraints": {
    "allowedPaths": [
      "C:\\...\\workspace02\\*",
      "C:\\...\\shared-projects\\*"
    ],
    "deniedActions": [
      "message.send",
      "sessions_spawn"
    ]
  }
}
```

- **allowedPaths**: 只能读写自己的 workspace + 共享目录
- **deniedActions**: 只有 PM 能 `message.send`（向人类汇报）和 `sessions_spawn`（创建新会话）；执行角色被禁止

**对比 Orchestrator**: 我们目前没有 Agent 级别的权限隔离。任何 sub-agent 理论上可以做任何事。

**可偷**: 在 Agent SDK 派发 sub-agent 时，通过 prompt 注入或配置文件限制可用工具集。特别是：
- 只有 Governor/PM 级 agent 能发 Telegram/外部消息
- 执行层 agent 只能读写指定目录

### P3: 四种会话类型分层通信 ⭐⭐

| 会话类型       | 用途          | 参与者    | 创建者  |
|---------------|---------------|---------|--------|
| 人→PM 私聊     | 任务分配、汇报 | 人、PM   | 人     |
| 项目群聊       | 状态同步、通知 | 所有代理 | PM     |
| 代理私聊       | 技术细节、协作 | 2 个代理 | 任意   |
| 代理独立会话   | 独立工作      | 单代理   | 系统   |

**私聊消息模板**:
```
【任务】XXX
  - 内容: XXX
  - 截止: XXX
  - 参考: projects/xxx/xxx.md

【确认】请确认 XXX
  - 问题: XXX
  - 建议: XXX
  - 请回复: 确认/修改意见

【提醒】XXX 已更新
  - 版本: v2
  - 变更: XXX
  - 请查看: projects/xxx/xxx.md
```

**对比 Orchestrator**: 我们的 Telegram channel 目前是平面的——所有消息走同一个管道。没有群聊/私聊/站会的分层。

**可偷**: 在 dispatch 系统中引入消息类型标签（任务/确认/提醒/站会），让 Agent 根据类型决定通信渠道和格式。

### P4: 三层 PM 审核模型 ⭐⭐⭐⭐

这是最有价值的 pattern。

```
┌─────────────────────────────────────────────┐
│ 第一层: 快速扫描（每日, ~500 tokens）        │
│ - 读取各代理汇总报告 (summary.md)            │
│ - 检查共享目录文件更新时间                    │
│ - 识别风险标记                                │
│ - 输出: 风险清单、需深入检查的代理            │
├─────────────────────────────────────────────┤
│                    ↓                         │
│ 第二层: 风险驱动检查（按需, ~2000-5000 tokens）│
│ - 高风险代理 → 读取所有输出文件               │
│ - 状态异常 → 抽样检查关键文件                 │
│ - 正常代理 → 信任汇总                         │
│ - 输出: 验证结果、问题清单                    │
├─────────────────────────────────────────────┤
│                    ↓                         │
│ 第三层: 记忆压缩 + 会话轮换（Token > 700K）  │
│ - 压缩当前会话记忆到 project-baseline.md      │
│ - 开启新会话继续                              │
│ - 新会话读取 baseline + 增量变更              │
│ - 输出: 归档记忆、新会话基线                  │
└─────────────────────────────────────────────┘
```

**审核代码**:
```python
def weekly_review():
    # 第一层: 快速扫描
    daily_result = daily_check()

    # 第二层: 风险驱动检查
    findings = {}
    for risk in daily_result["risks"]:
        if risk["status"] in ["delayed", "high_risk"]:
            findings[risk["agent"]] = deep_check(risk["agent"])

    # 第三层: 检查是否需要会话轮换
    if get_session_tokens() > 700000:
        compress_memory("memory/project-baseline.md")
        notify_pm("建议开启新会话继续")

    # 生成报告
    report = compile_report(daily_result, findings)
    send_to_boss(report)
```

**Token 预算表**:

| 任务       | Token 预算   | 触发条件          | 频率   |
|-----------|------------|----------------|--------|
| 每日扫描   | ~500       | 每天 9:00       | 每日   |
| 风险检查   | 2000-5000  | 发现风险/延迟     | 按需   |
| 周度审核   | 10000      | 每周五 17:00     | 每周   |
| 里程碑审核 | 50000+     | 阶段完成         | 每阶段 |
| 记忆压缩   | 5000       | Token > 700K   | 按需   |

月度预估: ~100K-200K tokens

**对比 Orchestrator**: 我们的 Governor eval pipeline 已经有分层检查的雏形，但没有这么明确的 token 预算和风险驱动的选择性深检。

**可偷**:
1. **Governor 的 daily scan 加 token 预算上限** — 快速扫描 < 500 tokens，只看变化摘要
2. **风险驱动的选择性深检** — 正常部门信任汇总，出问题的才展开全读
3. **会话轮换的 baseline 机制** — 长对话压缩为 baseline.md + 增量

### P5: 监理角色独立审计 ⭐⭐

子代理调度页面引入了三层角色权限：

| 角色     | 代理       | 职责                            | 权限                  |
|---------|----------|-------------------------------|----------------------|
| 项目经理 | work01   | 接收主公指令、任务拆解、资源分配 | 高 - 可调配执行员工     |
| 项目监理 | work02   | 独立监控、质量检查、风险预警   | 中 - 直接向主公汇报     |
| 执行员工 | work03-10 | 执行具体任务、进度汇报         | 低 - 向项目经理汇报     |

关键: **监理独立于 PM**，直接向人类老板汇报。PM 管不了监理，监理也不干预执行——纯审计角色。

**对比 Orchestrator**: 我们的 Governor（都察院）已经是独立审计角色，但目前它的检查是被动触发的。OpenClaw 的监理是主动的——有自己的检查清单和汇报节奏。

**可偷**: 让都察院按周期性节奏主动巡检，而不是等 Governor 被调用时才检查。

### P6: 符号链接数据共享拓扑 ⭐

```
workspace01/           # PM 私有
  ├── notes/           # 不共享
  └── projects/        # ⚠️ 符号链接 → shared-projects/{项目名}

workspace02/           # Hardware 私有
  ├── notes/
  └── projects/        # ⚠️ 符号链接 → 同一个 shared-projects

shared-projects/       # 唯一数据源
  └── {项目名}/
       ├── 01_需求分析/
       ├── 02_电气设计/
       └── ...
```

**对比 Orchestrator**: 我们用数据库（SQLite）作为共享状态。OpenClaw 用文件系统 + 符号链接更简单粗暴但对 LLM 更友好——Agent 可以直接 Read 文件而不需要 SQL 查询。

**评估**: 对我们来说不太适用。数据库方案更结构化，符号链接适合文档项目但不适合实时数据流。

### P7: 协作流程模板化 ⭐⭐

4 阶段项目协作流程：

1. **立项 (Day 1 上午)**: PM 创建群聊 → 发布需求 → 各代理确认任务
2. **并行执行 (Day 1-3)**: 每日站会（群聊）→ 各代理独立执行 → 阻塞问题私聊协调
3. **深度协作 (Day 3 下午)**: 跨角色技术讨论（HW↔SW 确认 IO、Proc↔HW 确认替代件）
4. **汇总汇报 (Day 5 周五)**: PM 收集周报 → 编译汇总 → 向老板汇报

**每日站会格式**:
```
[PM] → 群聊
"【站会】2026-03-28
 请各位汇报昨日工作、今日计划、阻塞问题"

[Hardware] → 群聊
"【站会】2026-03-28
 - 昨日: 完成电机选型（伺服电机 3 台）
 - 今日: 绘制电气原理图
 - 阻塞: 无"
```

**可偷**: 给 Orchestrator 的定期采集加一个「站会」模式——不是采数据，是让各部门 Agent 汇报当前状态和阻塞。

### P8: Agent 推理链落盘 ⭐⭐⭐

OpenClaw 通过飞书流式输出让用户看到 Agent 的完整思考过程。但真正有价值的不是"看"——是**持久化**。

**他们的做法**: `reasoning: false` 让 Qwen3.5 把推理过程输出为正文 → 飞书流式推送 → 用户实时看到。体验好，但数据没留下来。

**我们可以做得更好**: 不推外部服务，直接把 Agent 的推理链写入本地 JSONL：

```jsonl
{"ts":"2026-04-06T10:00:00Z","agent":"procurement","phase":"thinking","content":"BOM 清单有 3 个型号需要询价..."}
{"ts":"2026-04-06T10:00:01Z","agent":"procurement","phase":"action","tool":"read","target":"shared-projects/02_电气设计/BOM清单.xlsx"}
{"ts":"2026-04-06T10:00:03Z","agent":"procurement","phase":"thinking","content":"西门子 6ES7 价格偏高，查替代件..."}
{"ts":"2026-04-06T10:00:05Z","agent":"procurement","phase":"output","file":"03_采购/供应商报价单.xlsx"}
```

**三重价值**:
1. **可审计**: Governor 回溯任何 Agent 的决策链——不是看结果对不对，而是看推理过程有没有逻辑漏洞
2. **可训练**: 积累的推理链 = 免费的 eval / fine-tune 数据集。好决策是正例，坏决策是负例
3. **零成本**: 本地 JSONL 写入，不依赖外部 API，不消耗额外 token

**对比 Orchestrator**: 我们已经有 `run-log.jsonl`，但只记录了工具调用结果，没有记录 Agent 的推理过程。需要把 thinking chain 也落盘。

**可偷**: 在 Agent SDK 的 dispatch 层，hook `onThinking` / `onToolUse` / `onOutput` 事件，写入结构化 JSONL。Governor 的审核可以直接 grep 这些日志而不需要重新询问 Agent。

## 偷师优先级排序

| 优先级 | Pattern | 对 Orchestrator 的价值 | 实施难度 |
|--------|---------|---------------------|--------|
| P0 | P4: 三层审核模型 + Token 预算 | Governor 效率提升 10x | 低 |
| P0 | P1: 结构化角色约束文件 | 六部 prompt 可操作性大幅提升 | 低 |
| P0 | P8: Agent 推理链落盘 JSONL | 审计+训练数据零成本积累 | 低 |
| P1 | P2: Agent 沙箱权限 | 安全隔离，防止 sub-agent 越权 | 中 |
| P1 | P5: 监理角色主动巡检 | 都察院从被动变主动 | 中 |
| P2 | P3: 消息类型标签 | 通信协议更清晰 | 低 |
| P2 | P7: 站会模式 | 周期性状态汇报 | 低 |
| P3 | P6: 符号链接数据拓扑 | 不适用，已有更好方案 | - |

## 总评

OpenClaw 不是技术创新——它的底座就是 Claude Code，没有自研框架。但它的**系统设计思维**值得偷：

1. **角色建模精确**: 不是泛泛的 "你是一个 XX 助手"，而是精确到每种场景走哪个沟通渠道、输出写到哪个目录。这种精度在 LLM 时代特别重要——模糊的指令 = 模糊的输出。

2. **审核分层节省 Token**: 不是每次都全检，而是快速扫描 → 风险识别 → 选择性深检。月度 Token 消耗可控在 100K-200K，这在 Qwen3.5 这种非顶级模型上非常重要。

3. **监理独立于 PM**: 这个设计直接映射了现实中的项目管理最佳实践——执行和监督分离。我们的都察院已经是这个角色，但可以更主动。

4. **限制**: 整个系统面向工控项目这个特定领域，多 Agent 协作基于 Claude Code 的 sessions_send + agentToAgent 功能。没有自己的调度引擎，依赖用户手动管理会话。对比我们的 Orchestrator 有 Docker 容器、数据库、采集器等基础设施，这个系统更像是一个精心设计的 prompt 工程作品。
